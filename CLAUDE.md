## Что это за проект

**VENA** — премиальный сервис выездной IV-терапии (капельницы на дом/в офис) в Москве.
Клиент заказывает капельницу, бригада (медсестра + врачебный контроль) приезжает к нему.
Ключевая фишка — личный кабинет с динамикой анализов «было → стало» под контролем врача.

Целевая аудитория: обеспеченные клиенты, ценящие время, приватность и результат.
Тон бренда: премиум, честность, «состояние с доставкой к вам». Не масс-маркет, не «лечение болезней».

## Что строим (scope первой версии)

Всё в одном приложении (монолит), три части:
1. **Лендинг** — витрина, продажа, приём заявок. Прототип: `prototypes/landing.html`
2. **Личный кабинет клиента (ЛК)** — запись, визиты, анализы, платежи, документы. Прототип: `prototypes/client-portal.html`
3. **Админ-панель** — заявки, бригады, клиенты, проверка анализов врачом. Прототип: `prototypes/admin-panel.html`

Отдельный инструмент-референс: `prototypes/biomarker-parser.html` — парсер PDF с анализами (ИИ).

## Технологический стек

- **Next.js** (React, App Router) — фронтенд + бэкенд-API в одном проекте
- **TypeScript** — типизация
- **Tailwind CSS** — стили (прототипы уже используют эти принципы)
- **Яндекс Облако** — вся инфраструктура (российские дата-центры, важно для 152-ФЗ):
  - **Managed Service for PostgreSQL** — база данных
  - **Object Storage** — хранение файлов (PDF анализов)
  - **Compute Cloud** — серверы для приложения и обработки медданных
  - Авторизация — своя (вход по SMS через российского SMS-провайдера) или Яндекс ID
- **ЮKassa** — платежи (эквайринг)
- **amoCRM** — внешняя CRM для операционки (заявки, клиенты, воронка). Интеграция через API.

### ВАЖНО — локализация медданных (152-ФЗ)
Все персональные и особенно **медицинские данные хранятся и обрабатываются только в РФ** (Яндекс Облако).
Это критическое требование, не отклоняться от него. Следствия:
- База данных, файлы анализов, любые ПДн — только в российском контуре Яндекс Облака.
- **Парсер биомаркеров** не должен отправлять медданные во внешние (зарубежные) ИИ-API.
  Для прода — обработка внутри российского контура (self-hosted решение) или жёсткий парсер под 1-2 лаборатории.
  Внешний ИИ-парсинг допустим ТОЛЬКО если данные обезличены или есть отдельное юридическое основание — решает юрист.

## Архитектурный принцип — монолит с чёткими модулями

Один репозиторий, НЕ микросервисы. Внутри — модульное разделение по контурам:

- **Операционный контур** → работает через amoCRM (заявки, клиенты, запись, воронка). Мы это НЕ пишем сами, а интегрируемся.
- **Медицинский контур** → наш уникальный код: парсер биомаркеров, проверка врачом, база динамики анализов. Медданные (152-ФЗ) держим изолированно, с прицелом на возможный вынос на отдельный (российский) сервер.

Связка контуров — по ID клиента: amoCRM хранит «кто клиент и что купил», наш модуль — «анализы и динамику».

Подробнее: `docs/architecture.md`

## Как работать с прототипами

В `prototypes/` лежат готовые HTML-макеты — это **референс дизайна и логики**, не финальный код.
При разработке: воссоздавай эти экраны как React-компоненты на Tailwind, сохраняя визуал и флоу.
НЕ импортируй HTML напрямую — пересобирай в компоненты.

Дизайн-система (взять из прототипов):
- Тёмная тема: фон `#0B1410`, панели `#13241D`
- Золото: `#C9A86A`, светлое золото `#E3C78D`
- Текст: `#F2EFE8` (основной), `#A8B5AD` (приглушённый)
- Шрифты: Fraunces (заголовки, цифры), Inter (текст)
- Скругления, мягкие тени, премиум-ощущение

## Ключевые правила и решения проекта

1. **Медданные — особая категория (152-ФЗ).** Анализы храним с согласием клиента, защищённо, **только в российском контуре (Яндекс Облако)**. Перед загрузкой PDF — экран согласия. Медданные не покидают РФ.
2. **Анализы всегда проходят проверку врачом** перед попаданием в динамику клиента (гибридный флоу: клиент загружает → парсер распознаёт → врач в админке подтверждает → появляется у клиента).
3. **Запись = «желаемый слот» с подтверждением оператора**, не автобронь. Клиент выбирает время → заявка со статусом «на подтверждении» → оператор подтверждает после проверки доступности бригады.
4. **Не строим то, что есть готовым.** Операционку (заявки, воронка) ведём в amoCRM, платежи — ЮKassa. Своё пишем только уникальное (медконтур, ЛК с динамикой).
5. **Честность бренда.** Никаких ложных медицинских обещаний. Составы капельниц утверждает меддиректор.

Полный список: `docs/decisions.md`

## Приоритет разработки

Делаем всё три части, но в разумном порядке:
1. Каркас проекта (Next.js + Tailwind + Яндекс Облако: Managed PostgreSQL, Object Storage, авторизация)
2. Лендинг (проще всего, приём заявок)
3. ЛК клиента (ядро ценности)
4. Админ-панель (проверка анализов + управление)
5. Интеграции (amoCRM, ЮKassa) — по мере готовности
