# Техническое задание — VENA

## Обзор

Веб-приложение для премиального сервиса выездной IV-терапии. Три части в одном приложении:
лендинг, личный кабинет клиента, админ-панель. Плюс интеграции с amoCRM, ЮKassa, парсер анализов.

---

## 1. Лендинг

Витрина и приём заявок. Прототип: `prototypes/landing.html`.

**Блоки:**
- Hero: «Энергия и тонус. С доставкой к вам»
- «Почему капельница, а не таблетки» (усвоение, скорость, точность)
- «Что доставляем» — 6 продуктов по мотивам (энергия, детокс, иммунитет, красота, ясность, по анализам)
- **Карточки продуктов** — по клику открывается модальное окно с составом, показаниями, ценой, процессом (см. прототип)
- «Почему выезд» (время, приватность, скорость)
- Калькулятор форматов (разовый / курс)
- Безопасность (отстройка от серого рынка)
- B2B (drip-day для компаний)
- Форма заявки → создаёт лид в amoCRM

**Функции:**
- Форма заявки (имя, телефон, программа) → отправка в amoCRM через API
- Адаптивность (мобильные — приоритет)
- Продуктовые модалки с составами (составы утверждает меддиректор — держать в конфиге/БД)

---

## 2. Личный кабинет клиента (ЛК)

Ядро ценности. Прототип: `prototypes/client-portal.html` (там все экраны и состояния).

**Разделы (навигация):**
- **Главная** — ETA бригады, повтор последнего, статус курса, динамика, врач, подготовка
- **Записаться** — программа → дата → желаемое время → адрес → оплата → согласие на процедуру
- **Мои визиты** — предстоящие/история со статусами (на подтверждении → бригада назначена → выполнен → отменён), перенос/отмена
- **Анализы** — загрузка PDF → парсер → на проверке врачом → динамика «было→стало» с целями
- **Платежи** — история, чеки, управление абонементом
- **Документы** — договор-оферта, согласия (медданные, процедура), медвыписки
- **Бонусы** — накопления, промокод, рефералы, подарочный сертификат, статус
- **Поддержка** — форма обращения (→ amoCRM), FAQ
- **Профиль** — данные, адреса, уведомления, безопасность

**Состояния:**
- Новый клиент (пустой ЛК с онбордингом)
- Активный клиент (наполненный)
- Вход / регистрация (SMS-код + согласия при регистрации)

**Ключевые флоу:**
- **Запись:** желаемый слот → заявка «на подтверждении» → оператор подтверждает (не автобронь)
- **Анализы (гибрид):** согласие на медданные → загрузка PDF → парсер распознаёт → «на проверке врачом» → после подтверждения врачом появляется в динамике
- **Оплата:** онлайн (ЮKassa) / при визите / бонусами

---

## 3. Админ-панель

Операционный центр. Прототип: `prototypes/admin-panel.html`.

**Разделы:**
- **Сводка** — KPI (выручка, визиты, загрузка, курсы), расписание дня, активность
- **Заявки** — входящие, назначение бригады, статусы
- **Бригады** — на выезде, доступны по районам, покрытие по округам (резерв 2×)
- **Клиенты** — база, статусы (Gold/абонемент/новый), история
- **Анализы на проверке** — очередь → врач сверяет распознанное с PDF, правит, подтверждает → в динамику клиента

**Важно:** на боевой части операционки (заявки, клиенты) может дублироваться/жить в amoCRM.
Наш уникальный экран — **проверка анализов врачом** (его точно пишем сами).

---

## 4. Парсер биомаркеров

Прототип: `prototypes/biomarker-parser.html`.

Извлекает показатели из PDF анализов (ИИ). Вход — PDF из лаборатории, выход — структура (показатель, значение, единицы, референс, статус). Поддержка сравнения «было→стало».

**Реализация:**
- В прототипе работает через встроенный ИИ-доступ. В проде — свой API-ключ + бэкенд-эндпоинт.
- **152-ФЗ:** передача медданных во внешний ИИ-API требует согласия и проверки юрисдикции. Возможно — обработка на своём (российском) сервере или жёсткий парсер под 1-2 лаборатории. Решение — за юристом.
- Результат всегда проходит проверку врачом (см. гибридный флоу).

---

## 5. Интеграции

- **amoCRM** — заявки, клиенты, воронка. Создание лида/контакта/сделки через API. Синхрон по ID клиента.
- **ЮKassa** — приём платежей, чеки, возвраты.
- **Лаборатории** (позже) — получение результатов по договору/API.

---

## 6. Нефункциональные требования

- Мобильная адаптивность (приоритет — телефон)
- Безопасность медданных (152-ФЗ): согласия, защищённое хранение, аудит доступа
- PWA-возможности (добавление на экран, web-push) — для «повтор без поисковиков»
- Производительность: быстрый лендинг (SEO важен — трафик из поиска)

---

## 7. Что НЕ входит в первую версию

- Мобильное нативное приложение (позже, PWA достаточно)
- Полная диспетчеризация/геораспределение бригад (на старте — вручную в админке/CRM)
- API-интеграция с лабораториями (на старте — загрузка PDF)
- Своя BI-аналитика (на старте — из amoCRM)
