# Архитектура — VENA

## Принцип: монолит с чёткими модулями

Одно приложение (один репозиторий), НЕ микросервисы. Причины: команда — один человек + Claude Code,
нагрузка на старте небольшая, скорость и простота важнее «модности». Внутри — чистое модульное
разделение, чтобы при необходимости можно было вынести модуль позже.

## Два контура

```
┌─────────────────────────────────────────────────────────────┐
│  ПРИВЛЕЧЕНИЕ                                                  │
│  Лендинг (форма заявки)                                       │
└───────────────┬─────────────────────────────────────────────┘
                │ заявка
                ▼
┌──────────────────────────┐      ┌───────────────────────────┐
│  ОПЕРАЦИОННЫЙ КОНТУР      │◄────►│  МЕДИЦИНСКИЙ КОНТУР        │
│  = готовое (amoCRM)      │ ID   │  = своё (наш код)         │
│                          │клиен-│                           │
│  • Заявки и лиды         │ та   │  • Парсер биомаркеров     │
│  • Карточки клиентов     │      │  • Проверка врачом        │
│  • Запись, напоминания   │      │  • База динамики анализов │
│  • Воронка продаж        │      │  • ЛК клиента (динамика)  │
│  • Распределение бригад  │      │                           │
│    (вручную на старте)   │      │  медданные (152-ФЗ)       │
│                          │      │  изолированно             │
│  + ЮKassa (платежи)      │      │                           │
└──────────────────────────┘      └───────────────────────────┘
                                              ▲
                                              │ PDF (клиент загружает)
                                   ┌──────────┴───────────┐
                                   │  Лаборатория (внешн.) │
                                   └──────────────────────┘
```

## Разделение ответственности

| Данные | Мастер-система | Кто пишет |
|---|---|---|
| Контакт клиента (кто, телефон) | amoCRM | ЛК при регистрации (через API) |
| Обращения в поддержку | amoCRM | форма ЛК / оператор |
| История покупок, сделки | amoCRM | оператор |
| Заявки на запись | amoCRM (сделка) | ЛК создаёт, оператор ведёт |
| Анализы, биомаркеры, динамика | наш медконтур | парсер + врач |
| Что показать клиенту в ЛК | наш медконтур | агрегирует из обеих систем по ID |

**Ключ связки — ID клиента.** amoCRM знает «кто и что купил», медконтур знает «анализы и динамику».
ЛК показывает клиенту и то, и другое, связывая по ID.

## Медицинский контур — особые требования (152-ФЗ)

Медданные (результаты анализов) — особая категория ПДн. Требования:
- Согласие клиента на обработку (экран перед загрузкой PDF)
- Защищённое хранение (не просто файлы в облаке)
- Возможный вынос на отдельный российский сервер (если внешний ИИ-парсинг/хостинг не пройдёт по юрисдикции)

Поэтому медконтур — изолированный модуль с чёткой границей. Физически вынести можно позже; код
изолирован с самого начала.

## Гибридный флоу анализов

```
Клиент в ЛК: согласие на медданные → загрузка PDF
        ↓
Парсер (ИИ) распознаёт показатели
        ↓
Статус «на проверке врачом» (клиент видит)
        ↓
Врач в админке: сверяет с PDF, правит, подтверждает
        ↓
Биомаркеры сохраняются → появляются в динамике ЛК клиента
```

## Флоу записи

```
Клиент в ЛК: программа + желаемый слот + адрес + оплата + согласие на процедуру
        ↓
Заявка → сделка в amoCRM (статус «на подтверждении»)
        ↓
Оператор: проверяет доступность бригады → подтверждает слот
        ↓
Статус в ЛК: подтверждён → бригада назначена → (визит) → выполнен
```

## Стек по слоям

- **Фронтенд + бэкенд:** Next.js (App Router) — SSR-лендинг, клиентские ЛК/админка, API-роуты
- **Инфраструктура:** Яндекс Облако (российские дата-центры — требование 152-ФЗ)
  - **БД:** Managed Service for PostgreSQL
  - **Хранилище файлов (PDF):** Object Storage
  - **Серверы:** Compute Cloud
  - **Авторизация:** своя (SMS через российского провайдера) или Яндекс ID
- **Платежи:** ЮKassa (эквайринг, чеки)
- **CRM:** amoCRM (внешняя, через API)
- **Стили:** Tailwind CSS

### Локализация данных (критично, 152-ФЗ)
Все ПДн и медданные — только в РФ (Яндекс Облако). Медданные не покидают российский контур.
Парсер биомаркеров в проде обрабатывает данные внутри РФ (не через зарубежные ИИ-API), либо
на обезличенных данных, либо по отдельному юридическому основанию — решает юрист.

## Путь роста (когда упрёмся)

- Медконтур → уже в российском контуре (Яндекс Облако); при росте можно выделить в отдельный сервис
- Диспетчеризация бригад → своя система с геораспределением
- Лаборатории → API-интеграция вместо загрузки PDF
- Мобильное приложение → когда PWA перестанет хватать

Всё это — потом, по мере реальной необходимости. На старте — монолит.
