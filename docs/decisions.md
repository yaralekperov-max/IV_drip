# Зафиксированные решения — VENA

Ключевые продуктовые и технические решения, принятые в ходе проработки проекта.
Claude Code: учитывай их при разработке, не пересматривай без явного запроса.

## Бизнес-модель

- **Независимый бренд VENA**, не франшиза.
- **Партнёрство с лицензированной клиникой** (гибрид: фикс + % выручки) вместо своей мед-лицензии на старте. Операционная компания — «фронт-офис» (организационные/информационные услуги), медуслугу оказывает клиника по своей лицензии.
- **Ценообразование:** разовый визит 18 000 ₽; курс 4 = 56 000 ₽; корпоративный drip-day 10 000 ₽/чел (от 8).
- **Позиционирование:** пустая премиум-ниша. Ценность/состояние + метафора доставки, НЕ «лечение болезней».

## Продукт и бренд

- **Тон:** «Энергия и тонус. С доставкой к вам». Премиум, честность.
- **Никаких ложных медобещаний.** Не использовать «разработано/проверено НИИ» без реального подписанного партнёрства. Научность — через реального врача-консультанта (к.м.н.) и доказательную медицину.
- **Составы капельниц** утверждает меддиректор. В прототипах — типовые плейсхолдеры с дисклеймером.
- **Похудение:** не делать якорным продуктом, не обещать снижение веса. Только честная «метаболическая поддержка».

## Медсёстры / бригады

- **Пул самозанятых (НПД)/ГПХ**, оплата за выезд, НЕ штат на старте.
- Распределены по округам Москвы, резерв 2×, 1-2 платных дежурных для день-в-день.
- **Gig-модель («Uber для медсестёр») отклонена** как несовместимая с премиум-качеством.
- «Маркетплейс медсестёр» — возможный отдельный бизнес ПОЗЖЕ, не параллельно (расфокус).

## Технические решения

- **Стек:** Next.js + TypeScript + Tailwind + **Яндекс Облако** + ЮKassa + amoCRM.
- **Инфраструктура — Яндекс Облако** (российские дата-центры). Выбрано вместо Supabase из-за требования 152-ФЗ хранить ПДн россиян в РФ. Компоненты: Managed PostgreSQL (БД), Object Storage (файлы/PDF), Compute Cloud (серверы).
- **Локализация медданных (критично):** все ПДн и медданные — только в российском контуре, не покидают РФ. Это определяет и парсер (см. ниже), и хранилище, и обработку.
- **Монолит**, не микросервисы (см. architecture.md).
- **Операционку не пишем сами** — amoCRM (заявки, клиенты, воронка). Своё — только уникальное (медконтур, ЛК с динамикой).
- **Нативное приложение — не на старте.** PWA (добавление на экран + web-push) достаточно. Telegram-бот отклонён (VPN в РФ).

## Личный кабинет

- **Запись = желаемый слот с подтверждением оператора**, не автобронь (на старте распределение бригад ручное).
- **Анализы — гибридный флоу:** клиент загружает PDF → парсер → проверка врачом → потом в динамику. Клиент не видит сырой результат парсера.
- **Динамика «было→стало»** — ключевая отстройка, которой нет у конкурентов.

## Парсер биомаркеров

- **ИИ-парсинг** (PDF → модель → структура) выбран из-за устойчивости к разным форматам лабораторий, в отличие от жёстких парсеров.
- **152-ФЗ (обновлено):** раз вся инфраструктура в Яндекс Облаке (РФ), медданные не должны уходить во внешние (зарубежные) ИИ-API. Для прода парсер работает **внутри российского контура**: self-hosted модель на Compute Cloud, либо жёсткий парсер под 1-2 лаборатории, либо ИИ-API только на обезличенных данных. **Финальное решение — за юристом.** Прототип `biomarker-parser.html` использует внешний ИИ только как демо — в прод в таком виде не идёт.
- На старте при малом потоке допустим ручной ввод показателей врачом.
- **Результат ВСЕГДА проверяет врач** перед публикацией клиенту.

## Юридическое (обязательно в проде)

- Медицинский + налоговый юрист до запуска.
- **Согласие на обработку медданных (152-ФЗ)** — перед загрузкой анализов.
- **Информированное согласие на процедуру** — перед визитом.
- Договор-оферта — при регистрации.
- Договор с клиникой-партнёром проверяет юрист (агентская схема, НДС, лимит УСН).

## Откуда берутся результаты анализов

Три способа: клиент сам загружает (старт), лаборатория передаёт по договору+согласию, API-интеграция (масштаб).
На старте — загрузка PDF клиентом + парсер + проверка врачом.
Стратегически — использовать лабораторное преимущество основателя (он работает в lab-testing).
